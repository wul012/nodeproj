# OrderOps Node V8 运行调试归档

## 本版目标

V8 给 operation intent 创建接口增加 `Idempotency-Key`：

- 第一次提交同一个 key：创建新 intent，返回 HTTP 201。
- 同一个操作者、同一份输入、同一个 key 再提交：返回旧 intent，HTTP 200，`idempotency.reused=true`。
- 同一个 key 换成不同输入：返回 HTTP 409 `IDEMPOTENCY_KEY_CONFLICT`。
- replay 会写入 timeline 事件：`intent.idempotency_replayed`。

这版仍然不访问 Java / mini-kv 上游。

- `npm run typecheck`：通过。
- `npm test`：7 个测试文件、27 个测试通过。
- `npm run build`：通过。
- `/health`：返回 `status=ok`。
- `/api/v1/runtime/config`：确认 `upstreamProbesEnabled=false`、`upstreamActionsEnabled=false`。
- 首页 HTML 包含 `intentKey` 和 `idempotency-key`。
- 第一次 `POST /api/v1/operation-intents`：
  - HTTP 201。
  - `idempotency.reused=false`。
- 第二次同 key、同输入提交：
  - HTTP 200。
  - id 相同。
  - `idempotency.reused=true`。
  - timeline 为 `intent.created,intent.blocked,intent.idempotency_replayed`。
- 同 key 换动作提交：
  - HTTP 409。
  - `error=IDEMPOTENCY_KEY_CONFLICT`。
- `GET /api/v1/operation-intent-events?type=intent.idempotency_replayed`：
  - 能查询到 replay 事件。

## 本版一句话

V8 解决了后台控制台常见的重复提交问题：同一请求可以稳定 replay，冲突复用会被拦住，并且 replay 行为可追踪。
