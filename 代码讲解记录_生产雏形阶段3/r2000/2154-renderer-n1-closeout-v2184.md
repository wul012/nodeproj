# v2184：让最后三个例外成为可执行规则

## Goal and Non-goal / 目标与非目标

v2183 已把最后十一份真正拥有标题、元数据、段落、消息和下一步文案的完整报告迁移到共享构建器，仓库里只剩三个未命中标准化标记的 `Renderer.ts` 文件。表面上看，最简单的收口方式是把它们也套进某个 builder，让原始未标准化计数从三变成零；但逐行阅读源码后可以确认，这三个文件根本不生产 Markdown。它们各自只有一个导出函数，函数接收 `profile`，再把两个下游子渲染器返回的 `string[]` 按固定顺序展开合并。标题、空行、列表前缀、缺省文案和字段格式全部由子渲染器负责。若强行迁移，新的抽象只会包住两个函数调用，既不消除格式重复，也不建立新的行为边界，反而让读者多跳一层。

因此本版的目标不是追求“原始数字好看”，而是建立两个同时成立的事实：第一，census 必须继续诚实报告仓库中确有三个文件未调用共享构建器；第二，N1 必须能机械证明这三项都属于计划允许的 `composition-only`，且除此之外没有第四个未迁移文件。最终数字采用四元组表达：总 renderer 为 245，命中标准化机制的为 242，经过审计的组合豁免为 3，非豁免未标准化为 0。非目标同样明确：不改变三个包装器源码，不改变任何公开 Markdown，不新增路由或能力，不消费 Java、mini-kv 的新鲜证据，也不把“实现完成”写成外部评审已经通过。

四元组还有一个维护上的好处：它区分“工程进度”与“目录事实”。以后若某个组合器被删除，总数和豁免数会同时下降；若某个组合器真正接入共享 helper，标准化数上升而豁免数下降；若新增普通 renderer 却忘记迁移，非豁免数会直接上升。三种变化在输出中具有不同形状，评审者不用翻提交历史也能判断发生了重构、标准化还是回归。单一的“剩余数量”无法提供这种诊断能力。

## Entry Points / 入口

开发者和 CI 的统一入口仍是 `npm run renderer:census -- --json`。区别在于 `package.json` 不再只调用一个无门槛的统计脚本，而是默认附带 `--max-unstandardized=3`、`--max-waived=3` 与 `--max-non-waived=0`。第一个门保留原有 shrink-only 语义，防止未标准化总数回升；第二个门把豁免数量封顶为三，避免团队以后把“加一条例外”当作廉价逃生口；第三个门最关键，它规定任何未被清单批准的 renderer 都必须为零。调用者即使只输入普通 census 命令，也不会意外绕过 N1 的完成条件。

脚本内部入口是 `buildRendererCensus()`。它先读取 `docs/plans/renderer-consolidation-waivers.json`，再枚举 `src/services/*Renderer.ts`。JSON 是规范事实源，Markdown 文件只是面向评审者的解释视图。这样选择是为了避免从表格文本中猜字段：机器读取结构化对象，人阅读清晰表格，测试负责证明二者没有漏项。测试入口 `test/rendererCensusScript.test.ts` 同时覆盖库函数调用与真实子进程 CLI，因此既能检查返回模型，也能确认退出码和错误信息确实会阻断命令，而不只是内存里算出一个布尔值。

## Response Model / 响应模型

旧响应只有 `totalRenderers`、`standardizedRenderers`、`unstandardizedRenderers` 和形状提示。它能回答“还有几个文件没命中标记”，却回答不了“其中多少经过批准”“批准依据是否仍成立”。新响应保留所有旧字段，避免历史脚本和证据失去可比性，同时增加 `waiverPolicy`、`waivedUnstandardizedRenderers`、`nonWaivedUnstandardizedRenderers`、`waivedFiles` 与 `nonWaivedUnstandardizedFiles`。原始三项没有被减掉或改名，评审者仍能看到真实代码形态；N1 的完成判断则明确读取非豁免计数零。

每个 `waivedFiles` 条目不仅回显文件、原因和评审检查，还附带 AST 实际解析出的导出函数名、参数名和子函数调用顺序。例如审批资料组合器声明先调用 approval-packet，再调用 signed-approval-capture；脚本会把源码实际调用序列与 JSON 数组逐项比较。仅仅“仍然调用两个函数”不够，调换顺序也会失败，因为 section 数组的顺序可能影响公开报告。这个响应模型让评审者能从一份 JSON 同时看到政策、声明和观测结果，而不是在三个文件与一张手写表之间来回推断。

输出中的 `unstandardizedFiles` 与 `waivedFiles` 有意同时存在，二者不是重复字段。前者保持旧版 census 的兼容接口，只陈述扫描事实与形状信号；后者携带批准原因及 AST 观测；`nonWaivedUnstandardizedFiles` 则是行动队列。正常关闭态下行动队列为空，但字段不会被省略，因此消费脚本无需区分“没有问题”与“程序忘了计算”。JSON 的字段顺序稳定，文件列表在读取目录后排序，也便于评审者保存两次输出做文本差异。

## Upstream Evidence / 上游证据

本版所需证据全部位于 Node 仓库。三个包装器的源码证明它们没有格式所有权；它们调用的六个子渲染器已经在早期批次中使用 `renderProfileEntrySections` 完成标准化；v2183 census 则给出迁移后的 242/245 基线。这里不需要读取 Java jar、mini-kv 可执行文件或历史 sibling fixture，因为豁免判定只涉及 TypeScript 源码结构，不涉及跨项目业务事实。计划中的并行关系因此是“Java 与 mini-kv 推荐继续”，Node 不构成它们的前置批准者。

证据链刻意分成规范与现实两部分。规范由 JSON 写出“允许哪个文件、为什么允许、应调用哪些子渲染器”；现实由脚本每次重新扫描目录、读取源码并解析 AST。若只提交 JSON，任何文件后续增加一行 Markdown 都可能继续借用旧理由；若只靠 AST 自动猜测，又无法表达为什么这种结构值得豁免以及评审者应检查什么。二者相交后，清单不能凭空批准不存在的文件，源码也不能在无人更新政策的情况下改变职责。

## Service Flow / 服务流向

一次 census 的控制流分为六步。第一步解析清单并验证 schema：版本必须为一，`waivers` 必须为数组，每项的文件、原因、评审检查都必须是非空字符串，分类只能是 `composition-only`，子调用数组不能为空且不能重复。第二步枚举 245 个 renderer，读取源码并按既有七种标记计算标准化状态，同时保留 h3、for、map、flatMap 四类规划信号。第三步把清单文件映射回真实记录；文件不存在立即失败，文件已经标准化也立即失败，因为这代表旧豁免已过期，应删除而不是长期挂账。

第四步调用 `inspectCompositionOnlyRenderer()`。这里使用 TypeScript 自己的 parser，而不是依赖正则匹配大括号或函数名。解析结果除 import 外必须只剩一个函数声明；函数必须具名导出、只有一个标识符参数、函数体只有一条 return；return 表达式必须是数组；数组每一项都必须是 spread element，spread 的内容必须是直接函数调用，且唯一参数必须原样转发入口参数。字符串、模板文本、对象、条件表达式、局部变量、循环或额外语句都无法通过。第五步比较 AST 得到的调用名和顺序与清单声明，之后才把原始未标准化集合切成 waived 与 non-waived。第六步输出 JSON 或文本，并依次执行三道最大值门。任一门失败都会设置非零退出码，使测试或 CI 真正变红。

失败顺序也经过刻意安排。清单 schema 先于源码扫描验证，因此字段拼错会得到直接的政策错误；文件存在性和标准化状态随后检查，可以把“路径过期”与“豁免应删除”分开；只有这些前提成立才进入 AST，避免用复杂语法错误覆盖更简单的账本错误。最后才计算数量门，因为一个结构非法的豁免不能先被计入合法配额。这个顺序让维护者看到第一条错误时就能采取确定动作，而不是从多个衍生告警中猜根因。

## Safety Boundary / 安全边界

这套机制最重要的安全边界是“不把豁免变成另一种标准化标记”。标准化依然表示代码实际复用了共享 builder 或既有 section helper；豁免只是说明某个文件没有格式职责，迁移它不会减少重复。响应中两类数字始终分开，原始三项也始终可见。这样既不伪造 245/245，也不会让 N1 因三个有意保留的组合器永久停滞。

第二条边界是豁免会自动过期。如果未来开发者把某个包装器迁移到 builder，脚本会发现它既命中标准化标记又仍在清单中，并要求删除旧项；如果开发者在包装器里加入标题字符串，AST 会在数组元素处拒绝；如果新增第四个纯组合文件，非豁免零门先失败，即使随后把它加入清单，豁免上限三仍会失败，必须通过正式计划收紧或重新设计，而不能静默扩容。第三条边界是本版没有碰业务文件，因此不需要用快照更新来制造通过，也没有修改任何 renderer 测试期望。外部 N1-close review 仍是独立状态，进度表只写“implementation complete; review pending”。

AST 规则同样保持保守：它只批准当前确实需要的最小语法，不试图识别所有理论上等价的 TypeScript 写法。例如先把子结果存入局部变量再返回，或通过条件表达式选择子数组，即使最终可能没有格式化，也会被拒绝并要求重新评审。这不是 parser 能力不足，而是让豁免面可理解、可预测。需要更复杂控制流的文件已经拥有业务决策职责，应进入正常标准化或单独设计，而不应借用“纯组合”标签。

## Test Coverage / 测试覆盖

聚焦测试共有七项。基线测试重跑真实目录并断言 245、242、3、3、0 五个核心数字，检查三个文件名严格等于批准清单，检查每项 AST 观测到两个子调用。文档同步测试读取评审 Markdown，确保每个规范文件名和统一 reviewer check 都真实出现，避免只更新 JSON 而让人工材料过期。结构负例构造一个在返回数组中直接加入标题字符串的伪 renderer，证明 parser 会拒绝“组合器偷偷拥有格式”的变化。

另外两项 ratchet 负例分别把原始上限和豁免上限设为二，真实启动 Node 子进程并期待非零退出；它们证明门不是测试里的模拟判断。第六项把默认上限三与调用者追加的上限二同时传入，确认最后一个显式参数优先，使后续版本能够通过普通 npm 命令继续收紧基线。最后一项执行完整 3/3/0 CLI 并解析 JSON，确认仓库在默认完成条件下通过。后续总体验证还包括 typecheck、lint、build、文档质量门和低并发全量测试。由于 v2183 已暴露单命令在本机 30 分钟内可能因聚合成本超时，本版仍会把超时与断言失败分开处理，必要时使用互斥分片，不通过放宽业务断言或编辑 fixture 获得绿色结果。

评审复现时不需要相信归档里的数字。先运行默认 census，确认命令本身带有三个上限；再任选一个清单文件，把 return 数组临时加入字符串，聚焦测试应在 AST 规则处失败；恢复后把 `--max-waived` 改为二，CLI 应以非零状态退出。最后检查 Markdown 中三条文件名和 reviewer check 均来自 JSON。这个顺序分别验证正常态、结构负例、数量负例和人机材料一致性，覆盖了本版新增机制的四个独立故障面。

## One-sentence Summary / 一句话总结

v2184 用结构化清单、TypeScript AST 和 3/3/0 ratchet 证明最后三个文件只是无格式职责的组合器，从而把 N1 实现状态收敛为可复现的零个非豁免遗留项，同时把最终评审权明确留在外部检查点。
