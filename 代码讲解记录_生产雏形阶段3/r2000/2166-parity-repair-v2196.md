# v2196 代码讲解：把环境差异从行为差异中分离

## Goal and non-goal / 目标与非目标

本版目标是修正 parity oracle 对证据输入的建模：同一证据指纹下，v2194 与 v2195 的完整响应必须逐字节一致；不同证据指纹必须先经过旧新双跑才可登记。非目标是统一仓库换行、改写历史归档、修改生产摘要算法或放宽业务断言。本版只改变测试如何选择旧版基线，不改变任何运行时输出。

## Service flow / 服务流程

验证流程从四次 Fastify injection 开始：测试保存两个 JSON profile 与四个原始 body；随后从 JSON 递归提取 digest、byteLength 及其字段路径，排序后生成 EvidenceFingerprint；`expectedForFingerprint` 用该指纹取得 v2194 四响应数组，未知指纹先抛错；最后状态、正文长度和完整 SHA-256 做深相等比较。无论在哪个 checkout，指纹选择发生在输出断言之前，且不会修改响应。

## 一、为什么本地全绿仍然会得到红色 CI

v2195 做的是纯内部改名。六个文件、二十四个导出和所有调用者只发生了可枚举的符号替换，旧版源码应用同一替换表后与新版源码逐字节相等。两条归档证明路由在当前工作树中也完成了旧版、新版双跑：固定同一时钟、读取同一证据根后，两个 JSON 和两个 Markdown 的长度与 SHA-256 全部一致。focused、全量 559 文件和真实 HTTP 进程 smoke 都通过，因此实现本身没有留下未闭合的导入或行为分支。

远端 run 29180385904 却在最后一个新增测试失败。失败日志提供了很强的定位信号：typecheck、lint、安全、归档、elegance 全过，558 个其他测试文件与 1702 个其他测试也全过；四个 parity 响应状态都是 200，正文长度分别仍为 15313、13438、14866、13030，只有四个 SHA-256 与本地预期不同。这不是典型的代码回归。若字段增删、文本变化或 JSON 排序变化，正文长度通常至少有一项变化；四项长度稳定而摘要全部变化，更像若干等长摘要值被上游输入替换。

归档证明的职责恰好解释了这一现象。响应不只是业务状态，它还列出历史文件的相对路径、存在性、byteLength 与原始 SHA-256，并把这些值再次放入归档验证摘要。也就是说，报告的输出字节是“代码 + 当前证据原始字节”的函数。只要 Git checkout 在某些文本上选择 LF 或 CRLF 不同，即使人眼看到的行内容一致，byteLength、文件摘要、上层归档摘要以及最终 HTTP 正文摘要都会变化。

## 二、最初的 parity oracle 混淆了两个合同

第一个合同是迁移合同：旧代码和新代码面对完全相同的输入字节时，必须给出完全相同的输出字节。这个合同是 v2195 真正需要证明的内容，因为改名不应影响运行时。第二个合同是证据合同：如果输入归档的原始字节不同，报告必须诚实地给出不同 byteLength 和 digest，不能为了跨平台表面一致而偷偷规范化。原测试把当前 Windows 工作树的一组输出摘要写成唯一常量，等于错误地要求第二个合同失效。

这也解释了为什么不能简单把 GitHub 日志里的四个摘要覆盖到测试中。那会让 Linux checkout 通过，却使当前 Windows 工作树失败；更糟的是，它仍然没有表达摘要为何不同。按 `process.platform` 分支同样不可靠，因为决定输出的不是操作系统名称，而是仓库历史、Git 配置、clean/smudge filter 与文件后来是否被工具重写。两个 Windows 工作树完全可能有不同换行组合，一个 Linux 容器也可能挂载已经转换过的文件。

本版因此把测试模型改成“先识别输入，再验证完整输出”。输入识别不看平台，也不读取 CI 环境变量。测试从两个 JSON profile 递归收集键名为 `byteLength` 或以 `digest` 结尾的字段，连同字段路径和值组成一百零四条信号。数组索引与对象路径都进入信号，信号排序后做 SHA-256，得到 EvidenceFingerprint。这个指纹描述了影响四个响应的归档字节状态，同时避免把整份业务对象复制成另一套 fixture。

## 三、三套基线是怎样取得的

三套基线不是从失败日志随手抄入。第一套来自当前主工作树，它包含仓库长期演化后形成的混合换行状态，指纹为 `a8ede468...ac44d`。第二套使用 `git -c core.autocrlf=false worktree add` 创建隔离 checkout，保留 Git blob 的文本字节，指纹为 `c5574c92...8e10`；其四个响应摘要与 GitHub 日志逐项相同，独立复现了 runner 环境。第三套使用 `core.autocrlf=true` 创建 Windows CRLF checkout，指纹为 `98e84ad8...625d`。

每个 worktree 都先运行 v2195 代码，再让 v2194 代码读取同一 worktree 的证据根。时钟统一固定为 `2026-07-12T00:00:00.000Z`，请求头、配置和 forced historical fallback 完全相同。三种输入下，v2194 与 v2195 的四个状态、长度和摘要都逐项相等。这样得到的表表达的是三次独立迁移证明，而不是三个被宽恕的失败值。临时 worktree 的 `node_modules` 只用 junction 复用主安装，验证后先删除 junction，再通过 `git worktree remove` 清理；没有复制或修改归档。

## 四、测试代码现在如何失败关闭

`SURFACES` 只保存四个路由表面及格式后缀，不再把某一环境的摘要与路由定义揉在一起。`EXPECTED_BY_FINGERPRINT` 以完整六十四位证据指纹为键，每个值仍是四个完整响应的状态、字节数和 SHA-256 数组。测试发出请求后，一方面保留原始 body 计算完整响应摘要，另一方面只从两个 JSON body 计算证据指纹。最后先通过 `expectedForFingerprint` 查表，再把四个实际结果与选中的旧版数组做深相等比较。

如果未来历史证据真的被改写、字段路径发生变化、某个 digest 消失，或者 checkout 出现第四种未评审的字节组合，指纹不会命中。`expectedForFingerprint` 会立即抛出包含未知指纹的错误，不会退化到只比长度、只比业务字段或任意一个已知响应摘要。第二个 focused 测试显式传入六十四个零，机械证明未知输入走抛错分支。这个负向测试很重要，因为表驱动 oracle 若用默认值兜底，就会把“可解释的环境分型”变成“任何新环境都放过”。

完整响应摘要仍然保留。修复没有把 digest 字段替换成占位符，没有在 Markdown 上做正则清洗，也没有把 JSON parse 后重新排序。这意味着在某个已登记证据指纹内，空格、换行、字段顺序、文案、状态值或摘要算法的任何变化都会改变四正文至少一项 SHA-256，并让测试失败。指纹只负责回答“旧版在这组输入上应当输出哪一套字节”，不负责削弱输出比较。

## 五、为什么没有修改生产换行或摘要算法

一个看似直接的方案是在生产代码读取文本时统一把 CRLF 转成 LF。仓库已有 `historicalEvidenceReportUtils` 做过类似规范化，但它服务的是跨工作区历史文本消费；当前三个归档证明明确记录本地归档的原始 byteLength 与 SHA-256，语义不同。把规范化搬进生产路径会改变现有 JSON、Markdown 和 digest，违反 v2195 的行为冻结，也会让报告不再证明磁盘上的真实文件字节。

另一个方案是新增仓库级 `.gitattributes` 并强制全部文本 `eol=lf`。只读 census 发现受关注扩展名下有八千多个 tracked 文件，其中一千六百多个当前含 CRLF，且大量位于不可重写的历史归档。仓库级 renormalize 会制造巨大的历史字节变更；只加属性而不重写 blob 又会让当前工作树与新 checkout 暂时处于不同状态。那可以作为单独的数据治理项目评审，但不适合作为一个测试 oracle 的紧急修复，更不能混入命名版本。

因此本版坚持两个不变项：生产报告继续忠实散列实际证据字节，历史 archive 与 fixture 一字不改。变化只发生在测试如何选择“同输入下的旧版基线”。这比强行制造跨环境同摘要更符合系统机理，也把将来是否统一仓库换行留给独立、可评估的决策。

## 六、版本治理与后续影响

v2195 tag 已经推送，不能为了让历史看起来一次成功而强制移动。v2196 作为独立 CI 修复版本记录 run 29180385904 的失败边界、复现步骤和新 oracle；它占用 E-track 最多五版中的第三版。这样 E-N3 共享归档 engine 只能在 v2197 进行，终局 closeout 最晚是 v2198，仍然不超过硬上限。

本版不修改 `src`，名称债务仍是 4562，verification family 仍是 98。它的工程价值不是继续压数字，而是修正证据模型：以后看到 parity 失败，可以先判断是未知 evidence fingerprint，还是已知输入下的 response hash 变化。前者要求审计证据字节与 checkout；后者才指向产品行为或渲染变化。错误分类从“CI 与本地不同”提升为两个可执行分支。

E-N3 若继续抽取三个归档验证器的公共 engine，这个 oracle 会成为关键保护。共享读取代码会直接触碰 byteLength、digest、BOM 与 JSON 解析路径；三种证据指纹下的完整响应基线能够证明 engine 没有偷偷规范化或重排数据。换言之，v2196 不是对 v2195 的被动补丁，它把下一次高风险去重所需的跨 checkout 安全网补全了。

## 七、评审与回滚要点

评审时先确认 diff 只有测试、计划、证据和讲解，不应出现生产服务、路由、fixture 或 archive 修改。再检查 fingerprint 收集器：只有字段路径和值进入排序信号，不能使用平台名；键匹配必须覆盖小写 `digest` 和各种 `...Digest`，byteLength 必须单独包含。三张输出表的每一项都应有状态、长度和完整摘要，不能只存摘要前缀。

回滚 v2196 会恢复单一工作树 oracle，并使 Linux CI 重现已知失败，但不会改变生产行为。若要删除某个指纹，必须先证明对应 checkout 不再可能出现；若要新增第四个指纹，必须重复本版的旧代码、新代码同根双跑，而不能只取一份当前输出。这个准入条件写进简报和负向测试，防止表在未来演变成无审计的例外清单。

## 八、为什么指纹不直接散列整份 JSON

若直接把两个 JSON 正文再做一次 SHA-256 作为“输入指纹”，指纹与被验证的输出摘要会成为同一个东西。测试先用当前输出选择当前输出的期望值，逻辑上形成循环：业务字段或文案改变后会得到一个新指纹，只要有人把新值加入表就难以判断变化来自证据还是代码。EvidenceFingerprint 必须只描述外部输入差异，而不能把待验证行为吞进去。

本版选择 `byteLength` 与所有以 `digest` 结尾的字段，是因为 CI diff 与归档证明实现都显示，checkout 换行只通过这条因果链传播：历史文件字节变化，先改变文件长度和文件摘要，再改变重放对象中的来源摘要与归档验证摘要。字段路径也必须进入信号；如果只收集值，同一个摘要从 `source` 移到 `archiveVerification` 仍可能生成相同集合，掩盖结构变化。数组索引同理，能够发现 digest 记录被重排。

一百零四不是人为设定的阈值，而是当前两个 profile 递归扫描得到的实际信号数。测试不写死“必须是一百零四”来阻止合理字段演化，而是让任何增删自然生成未知指纹并失败。评审者看到未知指纹后，可以打印排序信号做差：若只涉及同一路径的 byteLength 与 digest，并能用旧新代码同根双跑复现，才有资格登记；若出现业务路径变化，就应回到产品 diff，而不是扩表。

## 九、修复后的故障分类怎样工作

第一类失败是 `Unregistered evidence fingerprint`。它发生在完整响应比较之前，说明测试看到一套尚未评审的证据字节。处理顺序是确认 checkout 配置、比较一百零四条信号、建立隔离证据根、运行旧版和新版。只有旧新四响应完全一致，才能把该指纹与旧版数组加入表。此过程不需要改产品代码，也不允许用当前新版单跑结果作为依据。

第二类失败是指纹已命中但四正文数组不相等。这时输入证据已经被控制，差异只能来自响应生成路径、字段排序、渲染、固定时钟失效或摘要算法变化。测试会显示具体是状态、长度还是 SHA-256 不同；维护者应先单跑对应路由和 renderer，再检查源码迁移，不能新增指纹，因为指纹本来没有变化。

第三类失败是 profile 无法产生预期的一百余条信号，例如路由返回错误对象、访问控制拒绝或 JSON 结构被删除。它通常也会形成未知指纹，但同时状态或正文长度会显著改变。focused 路由测试负责给出更直接的业务断言，而 parity 测试保留最终字节证据。两类测试互补：一个解释语义为何错，另一个证明输出确实不同。

通过这种分类，CI 不再只给出“Windows 绿、Linux 红”的模糊结论。每种失败都对应明确的下一条命令和责任边界，也避免维护者在压力下选择最省事但最危险的做法，例如改 fixture、重写 archive 或把新摘要无条件覆盖进期望。

## One-sentence summary / 一句话总结

v2196 用证据字节指纹选择经过旧新双跑验证的完整响应基线，使改名的字节等价证明既能跨 checkout 复现，又继续对未知输入和真实输出变化失败关闭。
