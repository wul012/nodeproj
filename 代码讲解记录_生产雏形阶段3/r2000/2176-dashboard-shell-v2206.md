# v2206：把 Dashboard 从“看起来能用”收口到结构正确、移动端可读

## 一、这一版从什么证据开始

这一版没有从“再找一个文件拆分”开始，而是直接承接 v2205 的真实浏览器结果。v2205 做的是大文件职责拆分，首要约束是页面字节完全不变。那次迁移虽然通过了摘要与完整 HTML 的长度、哈希校验，却在浏览器里暴露了三个迁移前就存在的问题：控制台出现 `/favicon.ico` 404；390 像素宽的页面里，少数英文按钮比自己的内容列更宽；完整 HTML 内部还出现了重复的文档壳标签。因为 v2205 的承诺是逐字节迁移，当时若顺手修复这些问题，就会把“结构重构”和“用户可见修复”混成一次不可区分的变化。因此 v2205 只记录发现，v2206 再用独立红测、独立字节基线和浏览器证据处理。

这条边界很重要。维护性重构的输入是旧输出，理想输出仍是旧输出；页面质量修复的输入则是已经证实的缺陷，输出应当与旧字节不同。两种工作如果共用一个模糊的“优化”标题，后来的人看到哈希变化时无法判断是合理修复还是重构漂移。本版计划先写明三项允许变化以及停止条件：文档壳只能保留一份，favicon 不再发起第二个请求，长按钮必须在 390 像素视口内完整可读。除此之外，路由、按钮动作、元素标识、业务 API、JSON、Markdown、fixture 和权限边界都不允许变化。

## 二、页面是怎样被拼出来的

Dashboard 不是模板引擎动态渲染，而是三个字符串模块按明确顺序组成。`dashboardHtml()` 是最外层装配函数，负责 doctype、`html`、`head`、viewport、title、style、`body`、脚本和结束标签；`dashboardMarkup` 是主体内容片段，包含页头、状态区、场景区和大量操作入口；`dashboardStyles` 是内嵌样式。可以把调用过程简化为：

```text
dashboardStyles ─┐
                 ├─> dashboardHtml() ─> HTTP GET / 的 text/html 响应
dashboardMarkup ─┘
```

问题出在职责边界没有完全贯彻。`dashboardHtml()` 已经输出 `<head>`，插入样式后再输出 `</head><body>`；但 `dashboardMarkup` 的开头也残留了一组 `</head><body>`。字符串拼接不会像严格模板编译器那样拒绝重复标签，浏览器的 HTML 解析器又会尽量容错，所以肉眼看页面仍可工作。容错恰恰让问题更隐蔽：源码的所有权错误被浏览器自动修补，测试若只搜索某个按钮或脚本片段，也不会失败。

本版新增的结构测试不依赖浏览器猜测，而是直接数完整页面中关键 token 的出现次数。doctype、`html`、`head`、`body` 的开标签和闭标签都必须各出现一次，同时 `dashboardMarkup` 必须完全不匹配这些文档级标签。这样测试表达的是所有权规则，而不是某一行当前写法。以后有人再次把壳标签塞回片段，即使页面仍能显示，测试也会立即失败。实现本身反而很小，只删除片段开头的 `</head>` 和 `<body>`；小实现之所以值得独立版本，是因为它背后新增了可以长期失败的结构契约。

## 三、为什么 favicon 选择内嵌数据，而不是新增静态文件

浏览器加载一个没有图标声明的页面时，通常会尝试请求站点根目录下的 `/favicon.ico`。OrderOps 没有为这个地址注册资源，于是 v2205 的控制台得到一次 404。单看功能，这不会阻止 Dashboard 工作；但它会污染浏览器控制台和网络面板，让真正的静态资源或 API 失败更难被第一眼发现，也意味着每次打开页面都产生一次毫无价值的请求。

这里有三种常见处理方式：新增静态文件与静态目录服务、增加一个专用 favicon 路由、或把小图标直接放进 HTML。前两种都会扩大运行时资源面或路由面，还需要处理路径、缓存、构建复制和归档；本项目的 Dashboard 是一个自包含运维页面，图标本身只需表达身份，不值得引入新的发布资产。于是本版复用外层 head 的元数据职责，加入经过百分号编码的 SVG data URL。图形只包含蓝色圆角底和白色环形标记，没有脚本、外链字体或远端资源。

测试先要求完整 HTML 包含 `rel="icon"` 且 `href` 以 `data:image/svg+xml,` 开始。这不是只测视觉颜色，而是锁定“不需要第二次 HTTP 请求”的策略。随后 Playwright 的网络记录提供运行时证据：页面加载产生一个根页面请求和十三个既有业务 API 请求，共十四个请求，没有 `/favicon.ico` 或其他图标请求；控制台错误和警告均为零。源代码断言证明声明不会无意删除，浏览器网络记录证明真实浏览器按预期消费了声明，两层证据各自覆盖不同风险。

## 四、移动端按钮为什么会越界

按钮区使用 flex 布局并允许换行。旧规则给每个 `.row button` 设置 `flex: 0 0 auto`，三个数字依次表示不放大、不收缩、基础尺寸由内容决定。短文字按钮没有问题；但“Deployment Execution Verification Report”之类连续英文标签会把按钮的内容宽度推到父列之外。父列在 390x844 视口中只有 302 像素可用宽度，v2205 快照却观察到 346 到 362 像素的按钮。`flex-wrap` 只能把整个项目换到下一行，并不能在 `flex-shrink: 0` 时把项目压回父容器，所以按钮即使独占一行仍然溢出。

新规则把它改成 `flex: 0 1 auto`：按钮仍不会为了填满空白而任意放大，但当内容的理想宽度超过容器时允许收缩。`min-width: 0` 处理 flex 项默认最小内容宽度这一层限制；如果缺少它，某些长内容即使允许 shrink，仍可能被 min-content 尺寸卡住。全局按钮的 `max-width: 100%` 给出最终几何上限，保证按钮不能超过所属列。`overflow-wrap: anywhere` 允许特别长、没有天然空格的 token 在必要时断行；`white-space: normal` 明确保留普通换行行为。

这组规则没有使用 `overflow: hidden`、固定高度或省略号。运维界面的按钮名称往往区分“记录”“报告”“验证报告”等相近操作，截掉尾部恰好会隐藏最关键的语义。正确输出不是把溢出的内容藏起来，而是保持完整文字并增加高度。390 像素视口的实际快照显示，按钮容器位于 x=37、宽度 302 像素；原先最长的一批按钮现在宽度最大恰好 302 像素，高度从 40 增到 60 像素，文字完整换行。短按钮仍保持原有 40 像素高度并可在一行组合，说明修复没有粗暴地把所有控件强制成同一种尺寸。

## 五、测试先行如何约束一次“允许改字节”的版本

实现前只修改既有 `dashboard.test.ts`，加入三项断言。第一次运行共有五个用例，其中三个按计划失败：完整文档标签计数错误，页面中没有 data favicon，样式中仍是禁止收缩且缺少换行上限。这个红色阶段证明测试确实能抓住 v2205 的已知缺陷，而不是在实现之后编写一组无论如何都会通过的装饰性断言。

完成 HTML 与 CSS 修复后，这三项结构测试转绿；此时 v2205 的字节 parity 测试按预期失败，而且只失败在 Dashboard 的长度与 SHA-256，同文件中的三十四条迁移路由检查继续通过。这个失败不能简单理解为“测试过时所以改掉”。更新 oracle 前先满足了三个条件：缺陷由独立结构测试复现；实际修改只落在三个既有 UI 模块；新页面已经由构建产物重新计算长度与哈希。最终 `dashboardMarkup` 从 38137 字节减少到 38122 字节，因为删掉十五个多余字符；完整页面从 95724 增加到 96100 字节，因为新增 favicon 声明和收缩换行样式。新的 markup SHA-256 是 `61219ccfdf720b9fc62eef0ac07edb0d8ef7891be36494cb7438d68d574760c6`，完整 HTML SHA-256 是 `20cd81e4606f159be77f7586573e8bf8d207bdda3b520fdeb5c049387e80c95a`。

parity 用例名称也从“拆分前冻结”改成“冻结 v2206 有意页面壳结果”。名称本身提醒维护者：这不是仍在声称字节从未变化，而是把一次已经解释、已经验证的变化设为下一次重构的起点。以后若页面再次漂移，维护者必须重新说明业务理由，不能把哈希更新当成机械动作。

## 六、从输入到输出的完整验证路径

本版的静态输入有三类：v2205 浏览器发现、当前三个 UI 字符串模块、现有 Dashboard 与路由 parity 测试。转换过程是先把发现改写成会失败的断言，再调整职责边界和 CSS 约束，最后计算新字节 oracle。静态输出是一个单壳 HTML 字符串、一条内嵌图标声明和一套能在窄列内保留全文的按钮样式。

运行时输入是以禁用上游探测、禁用生产动作、禁用访问强制的安全配置启动构建产物。服务在 31206 端口接受六类 smoke 请求：`/health` 返回 200 和 93 字节，`/api/v1/metrics` 返回 200 和 444 字节，`/api/v1/runtime/config` 返回 200 和 2155 字节，real-read rehearsal 的 JSON 与 Markdown 分别返回 200 和 4572、4819 字节，根页面返回 200 和 96100 字节。这里同时覆盖基础存活、观测面、运行时配置、迁移后的双格式报告和实际页面，不用只凭单个健康接口推断整个应用正常。

浏览器输出则是两张最终截图、零控制台错误、一份请求清单和移动端可访问性几何快照。桌面视口 1440x900 用来确认总体布局没有因换行规则退化；移动视口 390x844 用来验证最窄目标场景。截图是供人直观看布局的证据，bounding box 是供机器和评审复核尺寸的证据，两者不能互相替代。验收后浏览器被关闭，受工具跟踪的前台服务被终止，端口监听数复核为零，没有留下后台 Node 测试树。

最终完整套件还验证了讲解本身。第一次执行中，前三片 108 个文件、317 个用例通过，第四片的文档质量门指出本文缺少显式“一句话总结”；这不是业务回归，也不是超时。仓库没有放宽阈值或修改测试，而是补入下方真实总结，再依次重跑失败文件和完整第四片，分别得到 2/2 与 93/93 通过。第五至第十六片随后全部通过，最终为 16/16 分片、567 个测试文件和 1729 个用例；独立 `vitest list --json` 也返回 567/1729。最终树中，仓库扫描器对本文给出 3352 个中文字符、9 个可扫描二级章节、最大章节 584 字、无重复和超大段落、合规分 100。这个过程说明文档门确实会失败，而不是只在计划里存在。

## 七、这一版明确没有做什么

v2206 没有新建路由、服务、renderer 或证据链，没有更改 Dashboard 按钮绑定的 endpoint 字符串，也没有改 JSON 与 Markdown 输出。维护性 census 仍是 89 个近上限文件、121 个长函数、238 个高复杂函数和 2 个运行时导入循环；这里保持不变是正确结果，因为页面壳修复不应借机放宽或重写全仓基线。路由文件上限仍必须保持 80，所有既有 elegance、family、源码体积、renderer、安全配置和归档保留门仍要从最终树执行。

Java 正在处理自己的核心错误边界切片，mini-kv 正在处理自己的 shard 测试架构切片。它们都不提供本版输入，Node 也不读取它们的未提交工作树，因此两边可以并行，不需要等待 Node 批准。本版也不启动真实 Java jar、mini-kv CLI 或 live capstone，因为页面壳与 CSS 不改变跨项目契约；为了“显得完整”而启动兄弟进程反而会扩大验证噪声和清理风险。

## 八、维护者以后怎样判断相似改动是否合格

如果以后需要调整 Dashboard，先判断改动属于纯迁移还是有意呈现变化。纯迁移应先冻结完整输出并保持哈希；有意变化应先用独立结构或交互测试证明旧行为有问题，再更新字节基线。涉及片段时，文档级标签只能由 `dashboardHtml` 拥有；涉及静态元数据时，优先评估能否继续保持页面自包含；涉及窄屏布局时，不能只看截图“好像没问题”，至少要检查父容器与最长子项的实际宽度。

这版最终追求的不是多写几行 CSS，而是让页面装配的职责、网络请求的来源、flex 收缩的机理和字节 oracle 的更新理由都能被下一位维护者复现。输入是三个具体浏览器缺陷，输出是三条长期机械约束和两类运行时证据；当这些约束继续存在时，同类问题不能悄悄回到代码里，这才是本次优化真正留下的价值。

## 九、一句话总结

v2206 用先红后绿的结构测试、明确更新的页面哈希和两个真实视口，把三个浏览器缺陷变成可长期阻止回归的页面壳契约，同时保持 API、跨项目证据与执行权限不变。
