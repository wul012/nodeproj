# Node v2204 代码讲解：让大文件、长函数、高分支和运行时循环进入同一份收缩账

## 目标与非目标

v2204 的目标是给维护性建立一个可复现、会失败、只能收紧的底线。此前仓库已经有八百行源文件硬门、命名预算和结构家族检查，这些门能阻止明显失控，却不能精确描述六百至八百行之间的近线文件，也不能看到一个文件内部是否由超长函数或高分支函数支撑，更没有检查相对导入形成的运行时循环。结果是工程师能够凭经验说某个模块“看起来重”，却无法用同一条命令证明债务是否真的下降。新门把四种信号放进一张账，并要求每次变化都能解释为已知债务收缩。

非目标同样明确。本版不依据分数自动重写业务代码，不给复杂函数贴上“错误”标签，也不声称当前九十一个近线文件已经足够优雅。它不增加业务路由、响应字段、归档链或 sibling evidence 消费者，更不会借维护名义开放写操作。基线允许存量继续存在，是因为一次性重构数百个历史模块会扩大行为风险；基线不允许新增、增长或陈旧条目，是为了让后续每一次触碰都朝更小、更清楚的方向移动。

## 命令入口与文件职责

维护者的公开入口是 `npm run maintainability:census`，它调用 `scripts/maintainability-census.mjs`。该命令文件只负责参数、基线读写、差异比较、文本或 JSON 输出以及退出码，不理解 TypeScript 节点细节。真正的代码扫描集中在 `scripts/maintainability-scan.mjs`，它导出固定阈值和 `scanMaintainability`。两层分开后，测试可以针对临时项目复用同一个扫描实现，未来增加输出格式时也不会把图算法与终端表现混在一起。

第三个关键输入是 `docs/plans3/maintainability-baseline.json`。它是数据，不是第二套算法：顶部固定三个阈值，下面按类别保存精确键和当前允许的最大值。测试文件 `test/maintainabilityCensus.test.ts` 不复制扫描器，而是构造极小临时 TypeScript 项目，通过真实 CLI 观察退出码和 JSON 响应模型。CI 在现有 elegance 与 family 门之后调用同一 npm 命令，因此本地、focused test 和 GitHub runner 使用的是一条执行路径。

## 扫描输入与规范化过程

`scanMaintainability` 从项目的 `src` 目录递归收集 `.ts` 和 `.tsx` 文件，忽略声明文件。每个文件只读取一次，文本用于物理行统计，TypeScript `createSourceFile` 生成的语法树用于函数和 import 分析。所有键都转换为相对项目根目录、使用正斜杠的路径，这一点对 Windows 尤其重要：如果基线记录反斜杠而 CI 记录正斜杠，同一文件会被误报为一条陈旧债务加一条新增债务。

行数口径是文件的物理行数，只忽略文本末尾换行产生的虚空行。它与原有八百行硬门保持同一尺度，也诚实计算注释、空白排版和内嵌模板占据的导航距离；维护者不能通过把内容换成注释或插入空行逃离文件规模事实。当前树扫描到一千二百四十六个源文件。阈值使用严格大于六百，因此恰好六百行的文件不进入历史债务，六百零一行开始受精确键和上限约束。

## 近线文件模型为什么独立于八百行硬门

原有八百行门是绝对安全边界，任何新文件越线都失败；v2204 的六百行门解决的是提前治理。一个七百九十行文件还没有违反硬门，但下一次正常需求几乎没有舒适余量，而且阅读者往往需要同时加载多项职责。将它纳入 shrink-only 基线后，新文件不能进入这个区间，存量文件也不能从七百行长到七百零一行。这样，八百行门负责“绝不更坏到失控”，六百行账负责“接近失控前就开始偿还”。

文件键是相对路径，指标是物理行数。比较时先建立基线映射：扫描结果存在而映射中不存在，就是 unknown；两边都有但当前行数更高，就是 grown；基线存在而扫描结果不存在，就是 stale。要求 stale 失败很重要，因为修复完成后若仍保留旧条目，未来同一路径重新膨胀时会继承旧额度。维护者必须在真实修复后刷新基线，让已经退出六百行区间的路径彻底失去豁免。

## 具名长函数如何定位真实职责膨胀

文件总体不大并不代表内部结构健康。扫描器遍历函数声明、方法、构造器、访问器以及绑定到变量或属性的箭头函数和函数表达式，只跟踪能够形成稳定名称的节点。匿名回调通常生命周期短，若把每个数组回调都写入基线，行号变化会制造大量噪声；具名函数则有明确维护入口，适合建立跨提交键。函数键由相对路径、所属类或对象以及函数名组成，起始行只作为诊断信息，不作为身份的一部分。

函数跨度依据语法节点起止位置换算成物理行。超过一百二十行时记录当前最大跨度。这个指标不判断业务是否正确，它表达的是读者需要同时保留多少局部状态、条件和中间变量。后续提取 helper 后，原函数可能下降到阈值以下；此时旧键必须从基线删除。若只是把函数整体搬到另一个新文件，原键变陈旧而新键未知，命令会同时报两类失败，迫使维护者证明这是职责改善，而不是债务换地址。

## 分支分数怎样衡量认知路径

高分支检查使用 TypeScript AST，而不是对源码字符串搜索 `if`。扫描器从具名函数节点向下遍历，对条件语句、循环、`case`、`catch`、条件表达式以及逻辑与、逻辑或、空值合并等会增加执行路径的结构计分。超过二十分的函数进入基线，当前树共有二百三十八个。这不是严格的圈复杂度学术实现，而是一个稳定、可解释、足以做只减不增约束的工程分支分数。

字符串搜索容易把注释、Markdown 模板里的单词或变量名误当分支，AST 则只处理真正语法节点。分数与函数跨度分开保存，因为二者代表不同风险：一个一百五十行的线性格式化函数可能很长但路径简单，一个八十行检查函数可能短却包含大量组合条件。修复时可以提取数据表、判定策略或早返回，但不能仅靠增加换行降低分数；扫描器读取的是节点种类，格式化不会改变结果。

## 运行时 import 图与类型边界

循环检查先从每个源文件收集相对 import 和 export-from，解析无扩展名、`.js` 映射回 TypeScript、显式 TypeScript 扩展和目录 `index`。包依赖不进入这张图，因为本门只负责仓库内部模块结构。得到邻接表后，扫描器使用 Tarjan 强连通分量算法：一个分量包含两个以上文件，或者单文件存在自环，就代表从任一成员沿运行时依赖能回到自身。分量内路径排序后拼成稳定键，避免遍历顺序改变基线。

显式类型依赖被有意排除。`import type`、整条 type-only export 以及只含 type-only specifier 的声明在编译后没有 JavaScript 边，因此不会造成模块初始化循环。若把它们算入，许多共享 DTO 会产生虚假闭环，维护者反而可能把准确类型改成重复接口来讨好指标。普通 import 即使当前只在类型位置使用，只要没有显式写出类型语义，仍按运行时边处理；这鼓励作者把边界意图写清楚，也与 TypeScript 输出行为保持一致。

## 基线为何同时拒绝新增、增长与陈旧

很多基线门只比较总数，这会允许“修掉一个、再新增一个”的债务置换。v2204 保存精确键，并对带数值指标的类别同时保存 ceiling。unknown 阻止新热点，grown 阻止既有热点继续扩大，stale 要求已修复额度被及时收回。运行时循环没有单一可比较数值，因此比较精确成员集合；改变循环成员会表现为旧分量陈旧、新分量未知，同样不能静默通过。

阈值本身也属于合同。CLI 读取 JSON 后逐项验证 schema version、三个固定 limit、数组形状、键唯一性和非负数值。如果有人把六百改成七百，或把复杂度二十改成更宽松数值，命令在扫描差异之前就失败。刷新命令只应在重构确实完成后显式运行，生成值来自当前扫描器，不接受手工把某个 ceiling 调大。这样“更新基线”是一项可审查的债务收缩提交，而不是绕过门的通道。

## CLI 响应模型与退出语义

文本输出首先给出四类计数，再列出最靠近上限的文件和最高分支函数，最后打印 ready 状态。`--json` 输出包含 schema、limits、counts、topHotspots、violations、baselineErrors 和 ready，便于 Vitest 与后续工具稳定消费。ready 只有在基线结构正确、每个类别都没有 unknown、grown、stale 时才为真；任一条件失败都会设置非零退出码，因此 GitHub Actions 无需理解报告内容就能阻断回归。

`--project-root` 和 `--baseline` 让测试把命令指向临时项目，`--refresh-baseline` 负责显式重建账本。默认路径仍固定为仓库的 plans3 基线，普通开发者只需运行 npm script。命令层将路径解析、JSON 读取错误和数据形状错误统一放入 baselineErrors，避免文件缺失或 JSON 损坏时抛出一段难以定位的堆栈，更不能在基线不可读时错误地把空账当成通过。

## 对抗测试如何证明门不是展示品

focused test 的第一项在真实仓库根目录运行 CLI，断言 ready 为真，并检查测量口径、`package.json` 与 Node Evidence workflow 都包含该命令和基线路径。其余测试创建临时 `src`：一个六百多行、内部同时包含长函数和二十二个分支的样本必须触发三类 unknown；已有文件从六百零一行增长到六百零二行必须出现在 grown；基线中虚构的文件、函数和循环必须全部成为 stale。

循环测试建立 `a.ts` 与 `b.ts` 的普通互相导入，期望 CLI 非零退出并给出排序稳定的分量键。紧接着使用显式类型导入和纯类型 re-export 建立相同源码形状，期望循环计数为零且整体 ready。另一项故意写入语法错误后请求刷新基线，断言命令失败且旧文件字节不变，避免损坏源码被固化成新账。最后一项篡改三个阈值，确认每个漂移都进入 baselineErrors。所有 fixture 都在 `finally` 中删除；测试没有修改生产 fixture 或期望字节，也没有留下临时项目。八项测试共同证明通过路径和关键失败路径。

## 安全边界、上游证据与并行关系

本版只读取 Node 自身的 `src`、本地 JSON 基线和工作流配置，不读取 Java 或 mini-kv 的新产物。它没有注册 HTTP route，没有构造新的 profile service，也没有改变身份、审批、凭据或执行开关。扫描器看到的是静态源文件和 import 语法，不加载被扫描模块，因此不会因为分析代码而执行模块顶层副作用。CI 也只运行只读扫描与临时 fixture 测试。

跨项目成熟度标签保持不变：Node 已有只读联合验证，但本维护版本既不需要新鲜 sibling evidence，也不触发 live capstone。Java 和 mini-kv 可以在各自仓库继续推进，Node 不是它们的前置批准者。若未来跨项目合同文件本身超过门限，它会像普通 Node 源码一样进入扫描；修复仍必须保持冻结证据和外部路径不变，不能为降低指标改写上游事实。

## 本版验证结果与下一步维护方式

真实仓库结果是九十一个近线文件、一百二十二个长函数、二百三十八个高分支函数和两个运行时循环，四类精确事实均与基线一致。`maintainability:census` 返回 ready；focused Vitest 一个文件八项全部通过；typecheck 通过；lint 保持零错误、零警告。完整 Vitest、build、全部静态门与 HTTP smoke 没有在中间版本重复运行，而是按计划留到 v2205 最终树统一验收，届时 v2203 与 v2204 的改动同样会被覆盖。

后续维护先运行普通 census。如果报告 unknown，应判断是否出现新职责膨胀或新循环；如果 grown，应在同一文件继续提取概念，而不是提高 ceiling；如果 stale，说明修复已经有效，应显式刷新并审查基线 diff。v2205 将先处理排名最前的 `statusRoutes.ts` 与 `dashboardMarkup.ts`，目标是让两者退出六百行基线，并用路由注册与 dashboard 字节级测试证明拆分只改变内部组织。修复之后，九十一个近线文件至少应降到八十九个，不能用两个新热点抵消成果。

## 一句话总结

v2204 没有把历史维护债务包装成“全部优秀”，而是用 TypeScript 语法树、运行时依赖图、精确收缩基线和七条对抗测试建立共同尺度，使新热点、旧热点增长、债务搬家、额度残留和阈值放宽都会机械失败，并为下一版真正减少两个高价值热点提供可复现起点。
