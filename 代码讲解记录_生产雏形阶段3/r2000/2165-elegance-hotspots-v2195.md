# v2195 代码讲解：从过程句子中提取归档证明概念

## 一、这一版真正解决的是什么

v2195 不增加接口、不改变分片执行能力，也不扩大任何生产权限。它处理的是一个已经会持续拖慢维护的结构问题：若干 TypeScript 文件和导出名称把完整的历史过程直接拼进了名字。最长候选超过二百个字符，读者必须从“托管审计、人工沙箱、凭据解析、Java 与 mini-kv、声明式操作员生命周期、运行时执行”等十余层背景词里，才能找到文件末尾真正承担职责的几个词。这样的名字表面上信息很多，实际上没有形成可复用概念。每次导入都要复制整段历史，重构工具、错误栈、评审页面和 Windows 路径也被同一串文字挤满。

本版没有把“短”当作唯一目标。优雅门要求名字不超过四十字符，是为了迫使代码说出稳定抽象，而不是鼓励无意义缩写。最终选出的两个概念是 `ArtifactIntakeArchiveProof` 与 `PacketStopArchiveProof`。前者表示“从既有归档重放并证明工件接收前检状态”，后者表示“从既有归档重放并证明执行包停止记录状态”。Java、mini-kv、人工沙箱和操作员生命周期仍然存在于路由、配置及业务字段中，但不再重复占据每个内部类型和文件名。换句话说，背景留在边界，职责进入名字。

## 二、为什么没有直接改 census 最上面的五个名字

机械 census 的原始前五名来自文本包比较前检链，长度都在二百一十六字符以上。若只看排序，它们显然最刺眼；但计划的约束不是“把最大数字减掉”，而是“选择没有外部钉住的最坏候选”。只读审计发现，这组模块的精确 `.ts` 文件名被当前模块目录作为数据写入报告。若重命名文件并同步目录，报告字节会改变；若重命名文件却保留旧目录字符串，目录又会指向不存在的当前文件。两条路都违反本计划的失败条件，因此这组名字被明确淘汰，而不是用 waiver 掩盖。

随后对 filename violation 按长度继续向下筛选，并把每个候选的完整文件名与 `src` 中的字符串字面量、fixture、脚本、计划、waiver manifest 和包导出面交叉检查。最终得到的前五个安全候选分别是工件接收证明的 renderer、types、loader，以及停止记录证明的 renderer、types。为了不留下半个概念，本版同时收掉排名第六的停止记录 loader，形成两个完整的三文件族。仓库是 private package，`package.json` 没有 exports；这些文件只通过仓库内部相对导入使用。路由字符串虽然很长，却是独立常量，模块重命名不需要触碰它们。

这一步很重要，因为“最坏”必须同时满足严重程度与可安全修改两个条件。前五个原始条目不是安全候选，继续向下寻找并不等于降低标准，而是遵守行为冻结。优雅治理若为了漂亮报表改变报告内容，就已经偏离了维护目标。

## 三、两个短概念怎样落到文件与类型边界

每个概念保留三种职责。主文件负责读取归档、解析历史证据、重放冻结输入、计算检查项与摘要；types 文件只描述这一份证明的内存结构；renderer 文件只把 profile 渲染为 Markdown。没有把三者塞进一个新文件，也没有额外创建兼容 shim。旧路径被彻底删除，因此调用者只能使用新概念，不会形成新旧两套入口长期并存。

工件接收族现在由 `artifactIntakeArchiveProof.ts`、`artifactIntakeArchiveProofTypes.ts` 和 `artifactIntakeArchiveProofRenderer.ts` 组成。公开给内部调用者的两个行为名称是 `loadArtifactIntakeArchiveProof` 与 `renderArtifactIntakeArchiveProofMarkdown`。后者恰好四十字符，是本版最长的新标识符。类型名称也从整句式历史描述改成 `ArtifactIntakeArchiveProofProfile`、`ArtifactIntakeArchiveSource`、`ArtifactIntakeArchiveReplay`、`ArtifactIntakeArchiveRefs` 等。名称短了，但 source、replay、references、checks、summary、message 等角色仍然直接可见。

停止记录族采用对称但独立的 `packetStopArchiveProof` 概念。这里的对称不是复制新模板，而是让同层职责使用同一种语言。主 loader、profile、来源引用、重放引用和 renderer 都能从名字判断用途。实现期间 census 一度显示减少三十一条债务，比设计预期多一条。追踪 diff 后发现，批量替换误命中了更早一个版本中同名的 `RuntimeExecutionPacketStopRecordArchiveReferences`。那个类型指向 `e/391`，而当前证明类型指向 `e/392`，二者虽然旧名相同，却不属于同一数据边界。保留额外改名会让两种不同引用都叫 `PacketStopArchiveRefs`。本版因此主动撤回这条“有利于数字”的越界变化，让最终下降回到可解释的三十条。

这次回撤体现了命名治理的核心：短名称只有在抽象真实时才有价值。若两个不同对象被压成同名，即使所有测试通过、census 数字更小，代码仍会变得更难理解。机械门负责报警，工程判断负责解释报警是否对应正确概念。

## 四、为什么改导出名称不会改变业务输出

六个旧文件名和二十四个旧导出名都属于 TypeScript 编译期结构。JSON 对象的属性名、状态字符串、版本字符串、路由路径、归档相对路径、摘要输入、Markdown 标题与段落都没有参与替换。调用链只是从“导入一段一百多字符的函数名”改成“导入一个三十字符左右的概念名”，函数体执行顺序没有变化：读取文件、解析证据、重放 profile、构造 checks、计算 digest、构造 summary、渲染 Markdown，仍然沿用原来的代码。

为了避免批量字符串替换越过边界，本版在替换后做了三层检查。第一层是旧模块基名搜索，只允许业务对象字段中仍存在的长状态名，不允许旧 `.js` 导入路径或旧导出名残留。第二层是 TypeScript 类型检查，它证明所有调用者、re-export 和类型引用已经闭合。第三层是两份既有归档验证测试，覆盖成功归档、缺失归档时的失败关闭、JSON 路由与 Markdown 路由，共六项测试。这些测试的既有期望一字未改。

但“能编译且断言通过”仍不足以证明字节等价，因此本版又为四个 HTTP 表面采集正文长度与 SHA-256：两个 JSON、两个 Markdown。第一次直接比较时，四个长度完全相同，摘要却不同。检查后发现响应带有 `generatedAt`，普通请求每次都会产生不同时间。固定时钟后，临时 HEAD worktree 与主工作树仍有差异；进一步检查归档引用摘要，发现新 worktree 按 Git 配置把若干 Markdown 检出为 CRLF，而主工作树历史证据是 LF。归档证明本来就把原始文件字节纳入摘要，因此这不是代码差异，而是输入字节不同。

最终比较同时控制了两个变量：旧版和新版都固定在 `2026-07-12T00:00:00.000Z`，并都从主工作树读取同一套历史证据。四个响应的长度和 SHA-256 逐项一致。这个证明没有停留在手工记录，新增的 `eleganceHotspotParity.test.ts` 固定相同时钟、强制历史 fallback、带完整访问头，通过 Fastify injection 对四个正文做完整摘要断言。其期望值取自重命名前的 v2194 HEAD，不是根据新版输出随意编写。以后任何字段顺序、空格、换行、文案、摘要输入或路由响应变化都会让测试失败。

## 五、baseline 为什么从 4592 精确降到 4562

本版删除六个超长文件名。两个主文件各删除 loader 与 renderer re-export 两条超长导出，两个 renderer 各删除一条超长函数导出，两个 types 文件合计删除十八条超长接口导出，总计二十四个 export violation。六加二十四等于三十，因此总债务从 4592 降为 4562；其中 filename 从 993 降为 987，export 从 3599 降为 3575。

结构 family 也提供了第二个视角。旧主文件都以 `Verification` 结尾，所以 `src/services:verification` 从一百个文件降到九十八个。renderer 和 types 仍然保持原职责，数量分别维持二百四十五与二百二十一。两个新主文件以 `Proof` 结尾，形成两个成员的 proof family，低于“三个相似文件必须先抽 engine”的阈值，因此没有新增受管债务。baseline 只在 focused 行为测试与字节证明通过后刷新；刷新前 census 明确以 `shrunkNameDebt` 和 `shrunkFamilies` 失败，刷新后才恢复 ready=true。这个顺序保证 baseline 更新是在确认真实收缩，而不是提前为工作树放行。

## 六、这一版刻意没有做什么

本版没有顺手抽取三个大型归档验证器的公共引擎。只读比较确实看到工件接收、停止记录和 live-read gate plan 三个主文件存在明显结构相似性，它们分别约六百五十、六百四十和六百行。这个信号值得进入 E-N3，但不应与命名迁移混在同一版本。若在一次提交里同时移动文件、改导出、重写算法边界并改变摘要构造路径，出现字节差异时就难以确定原因，也会让评审无法把命名收益与去重收益分别复现。

下一版若继续处理该族，必须先列出真正相同的行为：文件引用生成、可选文件读取、稳定摘要、检查计数、冻结证据重放和失败关闭；再把版本号、归档根、必需文件、状态字段映射等差异留在策略数据中。只有三个成员能够共享同一个无业务猜测的 engine，并且三组旧输出都能逐字节对齐，抽取才成立。若共同部分只是表面代码形状而业务分支彼此不同，计划允许跳过，而不是为了完成“去重”里程碑制造一个参数几十项的万能函数。

## 七、维护者以后怎样判断这次治理是否仍然有效

维护者不需要相信本文结论，只需运行三个机械入口。`npm run elegance:census` 检查 4562 的名称债务摘要和九十八个 verification 文件上限，任何新增、同数替换、未同步收缩或 family 增长都会失败。`npx vitest run test/eleganceHotspotParity.test.ts` 检查两个归档证明的四个完整响应字节。两份原有 archive-verification 测试则检查成功、失败关闭和路由语义。三类证据分别约束结构、字节和行为，不互相替代。

因此，v2195 的价值不是把长词换成短词，而是把一段不可复用的过程历史变成两个稳定概念，同时留下可失败、可复现、能解释误差来源的证据链。名字变短只是表象；真正的改进是调用者现在可以先理解“证明什么”，再按需要进入“证明如何生成”，而不必每次重新解析整条项目历史。

## 八、评审这次改动时应该看哪些地方

评审者可以先把文档与 baseline 放到一边，只看源码 diff 的三种变化。第一种应当是六个文件的纯 rename，Git 能识别出主体内容仍是同一文件；第二种应当是 import、re-export、参数类型和返回类型中的符号替换；第三种只允许新增独立的字节回归测试。若 diff 中出现对象属性、字符串常量、数组顺序、摘要输入、文件读取路径或路由注册路径变化，就说明实现越过了本版边界，即使测试暂时为绿也应退回检查。

随后再看 baseline diff。它只能出现名称总数、filename/export 分类、key digest 和 verification family 上限的收紧，不能出现新增 waiver，也不能把 renderer、types 或其他 family 的上限调大。刷新 baseline 的提交顺序同样重要：先看旧 baseline 对当前树报告收缩失败，再看 focused 行为与字节测试通过，最后才接受新 baseline。这样可以区分“代码真的变好后同步合同”和“先改合同让代码通过”。

若需要回滚，本版不涉及数据迁移或外部状态，回滚六个路径与对应符号即可恢复旧编译结构；路由和响应从未改变，不需要兼容窗口。不过回滚后必须同时恢复 4592 的旧 baseline 与删除新增 parity 测试，否则机械门会正确地把债务回升识别为失败。这个回滚关系也说明 baseline 不是统计附件，而是与源码一起版本化的治理合同。
