# Node v2211 归档说明

v2211 处理的是一组真实而集中的维护性债务：mini-kv v114、v115、v116、v117 四份非参与回执分别由四个超长构造函数解析。它们虽然服务于不同阶段，却重复完成同一套工作：读取 JSON、按失败关闭原则给字段默认值、复制共同的禁止执行边界，再用几十个串联条件计算 ready。旧实现正确，但四个函数分别长 231、264、256、152 行，复杂度为 180、201、194、100；修改一个共同字段时，维护者必须在四个位置人工保持键顺序、默认值和判断一致。

本版先在未改生产源码时冻结 v114-v117 四个完整 profile 的 JSON 与 Markdown。固定时间、强制历史 fallback、路径便携化后，一共得到八个唯一尺寸和 SHA-256；占位断言先精确失败，写入基准后通过。实现随后把字段读取改为有序声明，由 `mapReceiptFields` 统一执行字符串、布尔值和数值的类型检查与默认值处理；重复输出键会直接抛错，避免清单静默覆盖。`miniKvReceiptFields` 只描述四份回执的数据形状，`miniKvReceiptReadiness` 只描述版本判定和共同的非参与边界，原服务保留证据清单、公开签名和最终装配。

重构后八个冻结值逐字不变，四个原有 focused suite 与新 parity suite 合计 5 个文件、20 项测试通过；字段引擎的规范化、错误类型默认和重复键拒绝测试也通过。维护性 census 在刷新基线前只报告 9 条预期 stale，刷新后从 86/116/232/2 收紧到 85/112/228/2，未出现新文件、新长函数、新复杂函数或新循环。v117 所在服务退出近上限名单，四个构造函数同时退出长函数和复杂度名单。

本版本不修改 fixture、历史路径映射、profile 字段顺序、digest、route、renderer、安全布尔值或执行权限。Java 保持在已提交 v1875、独立脏树 v1876；mini-kv 保持在已提交 v1673、独立脏树 v1674。Node 只借鉴它们的共享 renderer 与声明式 manifest 思路，没有读写、构建、测试或启动兄弟项目。没有 HTML/UI 变化，因此不生成截图。完整 suite、build、HTTP smoke、四版 push 与最终 CI 按批次简报在 v2214 后统一执行，本文件不提前把这些 pending 项写成通过。
