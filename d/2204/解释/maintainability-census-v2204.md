# v2204 说明：把维护性判断从观感变成只减不增的机械门

## 本版解决的问题

仓库原有的 `source:size:census` 能阻止单个 TypeScript 源文件越过八百行硬上限，却不能回答六百至八百行区间里还有多少接近上限的文件，也不能识别一个文件内部过长、分支过多的函数，更不能发现相对导入形成的运行时闭环。人工审查可以指出这些问题，但没有稳定口径时，不同会话会得到不同数字，修掉一处债务也可能在另一处重新长出。

v2204 新增一个本地脚本门，不新增 HTTP 路由、服务或业务报告。扫描器使用 TypeScript 语法树统计文件物理行数、具名函数物理跨度和分支分数，并从相对导入构建运行时依赖图。命令层把扫描结果与精确基线比较：新键失败、同一键数值增长失败、已消失的旧键仍留在基线也失败，阈值被改写同样失败。因此基线不是免责清单，而是一张必须随真实修复收缩的债务账。

## 当前结果与失败语义

当前树包含 1246 个 TypeScript 源文件和 6622 个具名函数。门记录 91 个超过六百物理行的文件、122 个跨度超过一百二十物理行的函数、238 个分支分数超过二十的函数，以及 2 个相对运行时导入强连通分量。命令输出 `Ready: true` 只表示这些事实与已提交基线完全一致，并不把历史债务描述成优秀状态。

八个对抗测试分别覆盖真实仓库接线、新增三类热点、既有热点增长、陈旧基线、运行时循环、显式类型导入与纯类型导出不被误判、语法错误阻止刷新，以及阈值漂移。普通相对导入会被解析到 `.ts`、`.tsx` 或目录索引文件，再通过强连通分量识别闭环。后续修复热点时，开发者必须显式刷新基线并提交缩小后的账本，不能让陈旧条目悄悄留下。

## 验证与边界

`npm run maintainability:census` 已通过，focused Vitest 为 1 个文件、8 项测试全部通过，TypeScript 类型检查通过，ESLint 保持 0 错误和 0 警告。完整 Vitest、build、全部静态门和 HTTP smoke 按三版本计划集中到 v2205 最终树运行一次。本版没有可渲染页面变化，因此不制造没有审查信息的截图；没有修改 fixture、跨项目证据、生产权限或 sibling 合同，也没有触发 live capstone。Java 与 mini-kv 可继续独立推进。
