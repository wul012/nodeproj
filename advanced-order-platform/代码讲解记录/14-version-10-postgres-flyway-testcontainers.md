# 第十版讲解：PostgreSQL、Flyway 和 Testcontainers

第十版的目标不是继续堆业务接口，而是把项目从“开发时能跑”推进到“更像真实公司后端项目”。

核心变化是：

```text
Hibernate ddl-auto: update
 -> Flyway 版本化迁移脚本
 -> Hibernate ddl-auto: validate
 -> PostgreSQL profile
 -> Testcontainers PostgreSQL 集成测试
```

这一步的价值是：表结构不再由 Hibernate 启动时偷偷修改，而是由可审查、可进入代码评审的 SQL 文件管理。

---

# 1. `pom.xml`：新增数据库工程化依赖

第十版新增 Flyway：

```xml
<dependency>
    <groupId>org.flywaydb</groupId>
    <artifactId>flyway-core</artifactId>
</dependency>
<dependency>
    <groupId>org.flywaydb</groupId>
    <artifactId>flyway-database-postgresql</artifactId>
</dependency>
```

`flyway-core` 负责基础迁移能力。Spring Boot 3.5 使用的 Flyway 版本中，PostgreSQL 这类非内存数据库需要数据库专用模块，所以又加入了 `flyway-database-postgresql`。

第十版还新增 Testcontainers：

```xml
<dependency>
    <groupId>org.testcontainers</groupId>
    <artifactId>junit-jupiter</artifactId>
    <scope>test</scope>
</dependency>
<dependency>
    <groupId>org.testcontainers</groupId>
    <artifactId>postgresql</artifactId>
    <scope>test</scope>
</dependency>
```

`junit-jupiter` 负责把容器生命周期接入 JUnit 5，`postgresql` 负责启动临时 PostgreSQL 容器。

一句话总结：`pom.xml` 让项目具备了“版本化建表”和“真实数据库集成测试”的基础能力。

# 2. `application.yml`：默认 H2 也走 Flyway

第十版默认仍然保留 H2，方便本地快速启动：

```yaml
spring:
  datasource:
    url: jdbc:h2:mem:order-platform;MODE=PostgreSQL;DATABASE_TO_LOWER=TRUE;DB_CLOSE_DELAY=-1
    username: sa
    password:
```

这里最关键的是：

```text
MODE=PostgreSQL
 -> 让 H2 尽量模拟 PostgreSQL SQL 行为

DATABASE_TO_LOWER=TRUE
 -> 让未加引号的表名/列名按 PostgreSQL 习惯转小写
```

第一次调试时遇到过一个问题：

```text
Flyway 已经执行建表
Hibernate validate 仍然提示 missing table [inventory_items]
```

原因就是 H2 标识符大小写和 Hibernate 小写表名不一致。改成 `DATABASE_TO_LOWER=TRUE` 后，Flyway 建出来的表和 Hibernate 校验使用的表名就对齐了。

Flyway 配置：

```yaml
spring:
  flyway:
    locations: "classpath:db/migration/{vendor}"
```

`{vendor}` 是 Spring Boot 支持的数据库厂商占位符：

```text
H2
 -> classpath:db/migration/h2

PostgreSQL
 -> classpath:db/migration/postgresql
```

JPA 配置从 `update` 改成：

```yaml
spring:
  jpa:
    hibernate:
      ddl-auto: validate
```

这表示 Hibernate 只校验实体和表结构是否匹配，不再负责自动改表。

一句话总结：第十版让 H2 不再是“Hibernate 自动建表玩具库”，而是同样经过 Flyway 迁移的开发数据库。

# 3. `application-postgres.yml`：真实数据库 profile

新增 PostgreSQL profile：

```yaml
spring:
  datasource:
    url: ${DB_URL:jdbc:postgresql://localhost:5432/order_platform}
    username: ${DB_USERNAME:order_app}
    password: ${DB_PASSWORD:order_app}
  h2:
    console:
      enabled: false
  jpa:
    hibernate:
      ddl-auto: validate
  flyway:
    locations: "classpath:db/migration/{vendor}"
```

这里保留环境变量入口：

```text
DB_URL
DB_USERNAME
DB_PASSWORD
```

本地可以用默认值，部署或连接不同数据库时可以通过环境变量覆盖。

启动 PostgreSQL profile：

```powershell
mvn spring-boot:run -Dspring-boot.run.profiles=postgres
```

打包后启动：

```powershell
java -jar target\advanced-order-platform-0.1.0-SNAPSHOT.jar --spring.profiles.active=postgres
```

一句话总结：`application-postgres.yml` 是项目从内存库切换到真实 PostgreSQL 的开关。

# 4. `docker-compose.yml`：本地 PostgreSQL 配合环境

第十版新增：

```yaml
services:
  postgres:
    image: postgres:16-alpine
    container_name: advanced-order-platform-postgres
    environment:
      POSTGRES_DB: order_platform
      POSTGRES_USER: order_app
      POSTGRES_PASSWORD: order_app
    ports:
      - "5432:5432"
```

启动命令：

```powershell
docker compose up -d postgres
```

然后用 PostgreSQL profile 启动应用：

```powershell
mvn spring-boot:run -Dspring-boot.run.profiles=postgres
```

这时应用启动顺序是：

```text
连接 PostgreSQL
 -> Flyway 执行 db/migration/postgresql/V1__initial_schema.sql
 -> Hibernate validate 校验实体和表结构
 -> DemoDataInitializer 插入演示商品和库存
 -> Web 服务启动
```

一句话总结：`docker-compose.yml` 给本机真实 PostgreSQL 调试留好了入口。

# 5. `V1__initial_schema.sql`：显式建表

第十版新增两份迁移脚本：

```text
src/main/resources/db/migration/h2/V1__initial_schema.sql
src/main/resources/db/migration/postgresql/V1__initial_schema.sql
```

它们负责创建当前项目全部核心表：

```text
products
inventory_items
orders
order_lines
order_status_history
payment_transactions
outbox_events
inventory_movements
```

以订单表为例：

```sql
create table orders (
    id bigint generated by default as identity primary key,
    customer_id uuid not null,
    idempotency_key varchar(120) not null,
    status varchar(32) not null,
    total_amount numeric(12, 2) not null,
    created_at timestamp(6) with time zone not null,
    paid_at timestamp(6) with time zone,
    shipped_at timestamp(6) with time zone,
    completed_at timestamp(6) with time zone,
    refunded_at timestamp(6) with time zone,
    canceled_at timestamp(6) with time zone,
    version bigint not null,
    constraint uk_orders_idempotency_key unique (idempotency_key)
);
```

这段 SQL 对应 `SalesOrder.java`：

```java
@Entity
@Table(name = "orders")
public class SalesOrder {
    @Id
    @GeneratedValue(strategy = GenerationType.IDENTITY)
    private Long id;

    @Column(nullable = false)
    private UUID customerId;

    @Column(nullable = false, unique = true, length = 120)
    private String idempotencyKey;

    @Version
    private long version;
}
```

以库存流水为例：

```sql
create table inventory_movements (
    id bigint generated by default as identity primary key,
    product_id bigint not null,
    type varchar(40) not null,
    quantity integer not null,
    available_before integer not null,
    reserved_before integer not null,
    available_after integer not null,
    reserved_after integer not null,
    created_at timestamp(6) with time zone not null
);
```

对应 `InventoryMovement.java`：

```java
@Table(
        name = "inventory_movements",
        indexes = @Index(name = "idx_inventory_movements_product_created", columnList = "product_id, created_at")
)
public class InventoryMovement {
    @Column(name = "product_id", nullable = false)
    private Long productId;

    @Enumerated(EnumType.STRING)
    @Column(nullable = false, length = 40)
    private InventoryMovementType type;
}
```

一句话总结：迁移脚本把“实体应该长什么样”落成了数据库可执行的契约。

# 6. `OutboxEvent.java`：payload 明确为 text

第十版对 Outbox payload 做了一个小调整：

```java
@Column(nullable = false, columnDefinition = "text")
private String payload;
```

之前使用 `@Lob` 更偏 JPA 抽象，但不同数据库对 String LOB 的实际类型可能不一致。

这里的 payload 是 JSON 字符串，不需要数据库大型对象机制，用 `text` 更直接，也更容易和 Flyway 脚本保持一致：

```sql
payload text not null
```

一句话总结：Outbox payload 从“抽象 LOB”改成“明确 text”，减少跨数据库校验的不确定性。

# 7. `PostgresMigrationIntegrationTests.java`：真实 PostgreSQL 集成测试

测试类核心注解：

```java
@Testcontainers(disabledWithoutDocker = true)
@SpringBootTest(properties = {
        "order.expiration.enabled=false",
        "outbox.publisher.enabled=false",
        "spring.jpa.hibernate.ddl-auto=validate",
        "spring.flyway.locations=classpath:db/migration/{vendor}"
})
class PostgresMigrationIntegrationTests {
```

`disabledWithoutDocker = true` 表示：

```text
如果本机 Docker 可用
 -> 启动 PostgreSQL 容器并执行测试

如果本机 Docker 不可用
 -> 自动跳过这条集成测试
```

容器定义：

```java
@Container
static final PostgreSQLContainer<?> POSTGRES = new PostgreSQLContainer<>(
        DockerImageName.parse("postgres:16-alpine"))
        .withDatabaseName("order_platform_test")
        .withUsername("order_app")
        .withPassword("order_app");
```

动态注入数据源：

```java
@DynamicPropertySource
static void registerPostgresProperties(DynamicPropertyRegistry registry) {
    registry.add("spring.datasource.url", POSTGRES::getJdbcUrl);
    registry.add("spring.datasource.username", POSTGRES::getUsername);
    registry.add("spring.datasource.password", POSTGRES::getPassword);
}
```

测试内容：

```java
Integer appliedMigrations = jdbcTemplate.queryForObject(
        "select count(*) from flyway_schema_history where success = true",
        Integer.class
);
```

它验证 Flyway 确实在 PostgreSQL 上执行过迁移。

然后验证核心表数量：

```java
assertThat(tableCount).isEqualTo(8);
```

最后跑一条真实业务链路：

```java
CreateOrderResult created = orderApplicationService.createOrder("postgres-it-v10-001", request);
orderApplicationService.pay(created.order().id());
var refunded = orderApplicationService.refund(created.order().id());
```

并断言库存流水：

```java
assertThat(movements.stream().map(InventoryMovementResponse::type).toList())
        .containsExactly(
                InventoryMovementType.RESERVE,
                InventoryMovementType.COMMIT_RESERVED,
                InventoryMovementType.RETURN_COMMITTED
        );
```

一句话总结：这个测试证明“SQL 迁移脚本、Hibernate 实体、真实 PostgreSQL、核心订单链路”能一起工作。

# 本次讲解总结

第十版的关键不是多一个业务功能，而是让项目的底座更像真实工程：

```text
建表由 SQL 版本管理
实体只负责校验表结构
H2 和 PostgreSQL 各走自己的迁移目录
Docker Compose 提供本地 PostgreSQL
Testcontainers 提供真实数据库集成测试
```

一句话总结：第十版把项目从“Hibernate 帮我建表”推进到“数据库结构有版本、有脚本、有真实数据库验证入口”。
