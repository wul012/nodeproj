# mini-kv v95 命令与字符串工具拆分归档说明

## 本轮计划来源

本轮推进前已只读查看 Node 计划目录：

- `D:\nodeproj\orderops-node\docs\plans\README.md`
- `D:\nodeproj\orderops-node\docs\plans\v223-post-external-adapter-readiness-roadmap.md`

最新计划里 mini-kv 仍处于单项目验证加跨项目契约对齐阶段。本轮 v95 是在前一版命令契约/响应 formatter 拆分之后，继续做一轮合理的结构收敛，重点是把 `trim_copy` 和 `to_upper` 这类重复字符串工具抽到共享头文件，同时把命令契约、响应 formatter、运行时证据和测试里的版本标记同步推进到 v95。

## 本版目标

v95 做了两件相互独立但同一方向的事：

- 提取共享字符串工具，减少 `trim_copy` / `to_upper` 在多个 cpp 里的重复定义
- 把当前版本、归档、fixture 和证据记录推进到 `0.95.0` / `v95`

它不是新的命令入口，不是执行审批，不是 Node/Java 适配器，也没有新增 restore/load/compact 的自动化能力。`CHECKJSON`、`EXPLAINJSON` 和各类运行响应仍然只解释契约，不执行目标命令；write/admin/WAL/snapshot/restore 边界保持不动。

## 结构变化

`include/minikv/string_utils.hpp` 是这版新增的共享工具头，提供两个 inline 工具：

- `trim_copy(std::string_view)`：去掉首尾空白，返回新字符串
- `to_upper(std::string_view)`：返回大写副本

它被以下文件复用：

- `src/command.cpp`
- `src/command_contracts.cpp`
- `src/command_response_formatters.cpp`
- `src/client_history.cpp`
- `src/client_main.cpp`
- `src/line_editor.cpp`
- `src/wal.cpp`

这样做的意义很朴素：把跨模块重复的小工具收束到一个地方，避免下次继续在多个 cpp 里各写一份，也让后续拆 parser / completion / command dispatch 时更容易看清职责边界。

## 文件职责

`src/command.cpp` 继续负责命令分发和真实执行路径，但现在不再自己维护重复字符串工具。它直接使用共享的 `to_upper()` / `trim_copy()`，同时继续调用命令契约和运行响应 formatter。

`src/client_main.cpp` 和 `src/line_editor.cpp` 复用同一套大写转换逻辑，避免 CLI 交互和补全逻辑各自维护副本。

`src/command_contracts.cpp`、`src/command_response_formatters.cpp`、`src/client_history.cpp` 和 `src/wal.cpp` 也统一改用共享工具，说明这不是单点修补，而是一次真正的公共工具收口。

`CMakeLists.txt` 把项目版本推进到 `0.95.0`，配合 README、fixture、release manifest 和 runtime evidence 中的 `v95` / `cmake-build-v95` / `c/95/` 标记一起同步。

## 控制面如何理解关键字段

这版没有改变控制面语义，只是继续沿用前一版的契约口径。

`read_only=true`、`execution_allowed=false` 仍表示只读契约检查；`command_digest` 仍是稳定的 `fnv1a64:<hex>` 指纹，不是鉴权签名；`side_effects` 和 `side_effect_count` 继续描述真实执行时的风险面，不是执行记录。

这意味着：本版在结构上继续变整齐，但在行为上仍然保持“只读证据、边界清楚、不会偷偷变成执行入口”。

## 测试和真实运行

`tests/command_tests.cpp`、`tests/release_verification_manifest_tests.cpp`、`tests/runtime_smoke_evidence_tests.cpp` 等测试已经同步到 v95，主要保护三件事：

- 命令契约输出仍然兼容
- release manifest 仍记录正确的源文件和边界说明
- runtime smoke 仍然只是只读证据，不会越界执行 restore/load/compact

真实 smoke 也跑过一次：`minikv_server` 启动后，`minikv_client` 依次执行 `SMOKEJSON`、`INFOJSON`、`STORAGEJSON`、`HEALTH`、`GET restore:real-read-token` 和 `QUIT`。结果里 `GET restore:real-read-token` 仍然是 `(nil)`，说明这轮没有引入任何写入或恢复执行路径。
