# Java v110 运行说明：credential resolver fake-shell archive echo receipt

## 计划依据

本版依据当前唯一有效计划：

```text
D:\nodeproj\orderops-node\docs\plans\v266-post-fake-shell-archive-roadmap.md
```

计划要求 Java v110 只读消费 Node v266 的 credential resolver fake-shell archive verification，生成 Java 侧 echo receipt，证明 Java 仍不写 approval ledger、不执行 SQL、不读取 credential value、不打开 managed audit connection。

## 合理性判断

合理。Node v266 已经把 v264/v265 的 fake-shell 归档证据收口，Java 现在需要做的是上游可读的回显收据，而不是进入真实 resolver 实现。

这版同时执行了计划里的质量要求：没有把字段堆回 `OpsEvidenceService`，而是拆出 records、builder 和 archive support。主 service 只增加常量和装配入口。


## 拆分结果

本轮没有让新 builder 继续膨胀成一个 800 行大文件，最终拆成：

```text
ReleaseApprovalManagedAuditSandboxEndpointCredentialResolverFakeShellArchiveEchoReceiptBuilder.java  450 行
ReleaseApprovalSandboxEndpointCredentialResolverFakeShellArchiveEchoSupport.java                    427 行
ReleaseApprovalSandboxEndpointCredentialResolverFakeShellArchiveEchoRecords.java                    201 行
```

builder 负责回执装配、digest、warning/proof/hint；support 负责归档文件、snippet、check 和 side-effect boundary；records 只负责响应类型。


Java v110 可以让 Node v267 读取这个 receipt，但不能被当作真实 resolver、真实 credential、真实 endpoint 或 managed audit connection 的许可。

## 清理记录

本轮验证后删除 `target`，没有遗留 Java/Maven 后台进程。

## 一句话总结

Java v110 把 Node v266 的 fake-shell archive verification 变成 Java 侧只读 echo receipt，同时把新增逻辑拆在 builder/support/records 三个边界里，没有打开任何真实 resolver 或 managed-audit 执行面。
