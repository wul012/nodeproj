# Java v109 代码讲解：release approval rehearsal response records split

## 本版所处项目进度

最新计划仍来自：

```text
D:\nodeproj\orderops-node\docs\plans\v263-post-disabled-resolver-echo-roadmap.md
```

v109 不新增业务 marker，也不碰 managed-audit 边界，而是把 `ReleaseApprovalRehearsalResponse` 里那组大块 nested records 拆到独立的 `ReleaseApprovalRehearsalResponseRecords` 容器里，主响应文件只保留 68 行壳。

## 为什么现在做

v108 只收口了 echo marker support 的重复样板，但 `ReleaseApprovalRehearsalResponse` 本体仍然接近 1800 行，而且外部 builder 还在密集引用这些 nested record。继续往里堆，会让后面的任意新字段都更难维护。

这次拆分的目标不是“再瘦一点点”，而是把响应壳和 record catalog 分离：
- `ReleaseApprovalRehearsalResponse` 只负责顶层返回值
- `ReleaseApprovalRehearsalResponseRecords` 负责所有嵌套 record 定义
- 现有 builder / test 只改引用前缀，不改字段语义

## 本版改动

新增文件：

```text
src/main/java/com/codexdemo/orderplatform/ops/ReleaseApprovalRehearsalResponseRecords.java
```

收缩后的主文件：

```text
src/main/java/com/codexdemo/orderplatform/ops/ReleaseApprovalRehearsalResponse.java
```

结果是：
- 主文件从 1798 行降到 68 行
- record catalog 集中到单独容器，便于后续继续分层拆
- 业务字段、JSON 契约、managed-audit / SQL / ledger / auto-start 边界不变

## 合理性

合理。因为这次不是为了好看而切，而是把“返回壳”和“类型目录”分开，后续再加响应字段时，不必继续在同一个文件里堆长链 nested records。
